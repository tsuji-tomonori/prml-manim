## Scene 1: モデル選択とは

**画面**

左に点群と多項式曲線、右に「複雑さ M」のダイヤルを表示する。ダイヤルを上げると曲線が柔らかくなり、下げると硬くなる。

**ナレーション**

前回の多項式曲線フィッティングでは、次数 M を変えると曲線の自由さが変わりました。
M が小さすぎると、データの流れを追えません。
M が大きすぎると、訓練データに合わせすぎます。
では、どの M を選べばよいのでしょうか。
この「モデルの複雑さをどう選ぶか」が、モデル選択です。
目的は、訓練データで一番よく見えるモデルではなく、まだ見ていないデータに強いモデルを選ぶことです。

**原文参照**

PRML p.32 の Section 1.3 冒頭は、最小二乗の例で最良の汎化を与える多項式次数があり、次数が自由パラメータ数とモデル複雑さを支配すると説明している。

## Scene 2: 訓練誤差だけでは選べない

**画面**

横軸を M、縦軸を誤差にしたグラフを描く。訓練誤差は右下がり、初見データの誤差は U 字型にする。訓練誤差だけの最小点 M=9 と、初見データでよい点 M=3 付近を比較する。

**ナレーション**

訓練データだけを見ると、M を大きくするほど誤差は下がります。
曲線が自由になれば、見えている点には合わせやすいからです。
でも、初見データの誤差は同じ動きにはなりません。
最初は下がりますが、複雑にしすぎると、また上がります。
訓練誤差だけで選ぶと、つい一番複雑なモデルを選んでしまいます。
それは、練習問題に一番合わせたモデルであって、初見問題に一番強いモデルとは限りません。

**原文参照**

PRML p.32 は、maximum likelihood では training set 上の性能が over-fitting のため unseen data の予測性能を示すよい指標にならないと述べている。

## Scene 3: Validation set と Test set

**画面**

データ列を `train`、`validation`、`test` に分ける。複数の M 候補が train で学習され、validation のスコアで M=3 を選ぶ。test は最後までロックされている表示にする。

**ナレーション**

データが十分にあるなら、使い道を分ける方法があります。
まず train、訓練データで、候補のモデルをそれぞれ学習します。
次に validation、検証データで、どの複雑さがよいかを比べます。
そして最後に test、テストデータで、選び終わったモデルの実力を一度だけ測ります。
大事なのは、テストデータを選択の途中で何度も見ないことです。
何度も見ながら調整すると、検証データやテストデータにも合わせすぎてしまうからです。

**原文参照**

PRML p.32 は、validation set で複雑さを選び、限られたデータで設計を何度も繰り返す場合は、選択済みモデルの最終評価用に第三の test set が必要になることがあると説明している。

## Scene 4: Cross-validation

**画面**

データを4つのブロックに分ける。1回目は1ブロックを赤くして評価用にし、残りを学習用にする。赤いブロックが順に移動し、4つのスコアが平均される。

**ナレーション**

でも、いつもデータがたくさんあるとは限りません。
検証用に多く取り分けると、学習に使えるデータが減ります。
検証用が少なすぎると、評価がたまたまに左右されます。
この困りごとへの代表的な答えが、交差検証です。
たとえば四分割なら、四つのうち三つで学習し、残り一つで評価します。
評価に使うブロックを入れ替えて、これを四回繰り返します。
最後に四つの点数を平均して、複雑さを比べます。

**原文参照**

PRML p.32-33 と Fig. 1.18 は、S-fold cross-validation で S 個のグループを順に held-out group とし、S 回の性能スコアを平均する方法を説明している。

## Scene 5: Leave-one-out

**画面**

データを細い N 個のブロックに分け、1点だけを赤くして残りを学習用にする。赤い1点が横に動き、`S=N`、`leave-one-out` を表示する。

**ナレーション**

さらにデータが少ないときは、S をデータ数 N と同じにすることもあります。
これは、ひとつだけ取り分けて、残り全部で学習する方法です。
名前は leave-one-out。
学習にはほとんど全部のデータを使えます。
その代わり、データが N 個なら、N 回の学習が必要になります。

**原文参照**

PRML p.33 は、データが特に少ない場合に S=N とする leave-one-out technique が適切なことがあると述べている。

## Scene 6: 探索コスト

**画面**

横軸に多項式次数 M、縦軸に正則化 λ を置いたグリッドを表示する。候補セルが増え、さらに別のパラメータ軸が追加される様子を見せる。

**ナレーション**

交差検証には弱点もあります。
まず、S 分割なら、学習回数はだいたい S 倍になります。
さらに、調整したい複雑さのつまみが一つとは限りません。
次数 M、正則化の強さ λ、ほかの設定。
つまみが増えると、試す組み合わせはすぐに増えます。
重いモデルでは、この探索だけで大きな計算コストになります。

**原文参照**

PRML p.33 は、cross-validation では training run が S 倍に増え、複数の complexity parameter を探索すると最悪の場合パラメータ数に対して指数的に training run が増えると説明している。

## Scene 7: 情報量基準

**画面**

左に `best-fit log likelihood`、右に `complexity penalty` を置き、差し引きで AIC の式 `ln p(D|w_ML) - M` を作る。複雑すぎるモデルのスコアがペナルティで下がる様子を棒グラフで示す。

**ナレーション**

そこで、検証データを使わず、訓練データだけから選べないか、という方向も考えられます。
歴史的には、情報量基準と呼ばれる方法が提案されました。
AIC はその一つです。
よく当てはまるほど、対数尤度は大きくなります。
一方で、調整できるパラメータが多いモデルには、複雑さの分だけペナルティを入れます。
つまり、当てはまりの良さから、複雑すぎる分を差し引く考え方です。

**原文参照**

PRML p.33 の Eq. (1.73) は、AIC が `ln p(D|w_ML) - M` を最大にするモデルを選ぶと説明している。

## Scene 8: 1.4 への橋渡し

**画面**

`train/validation/test`、`cross-validation`、`information criteria`、`Bayesian approach` の4項目を整理し、最後に高次元グリッドへ変形して次節へつなぐ。

**ナレーション**

モデル選択の中心にある問いは一つです。
見えているデータへの当てはまりと、まだ見ていないデータへの強さを、どう区別するか。
検証データを分ける方法。
交差検証でデータを使い回す方法。
複雑さのペナルティを入れる方法。
そして、PRML がこの先で重視する、ベイズ的な方法。
どれも、過学習の楽観をどう抑えるか、という問題への答えです。
次は、入力の数が増えたときに何が起きるのか、次元の呪いへ進みます。

**原文参照**

PRML p.33 は、理想的には training data のみに依存し、over-fitting による bias を受けない性能尺度が必要だと述べ、fully Bayesian approach へつなげている。
